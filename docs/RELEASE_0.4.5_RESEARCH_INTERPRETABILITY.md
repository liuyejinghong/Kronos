# Kronos v0.4.5 版本需求：历史重放、市场状态分段与只读观察边界

> 状态：已完成首版实现
> 版本目标：0.4.5
> 约束来源：`TODO.md`、`docs/PROJECT_STATUS.md`、`docs/ROADMAP.md`、`docs/PRODUCT_DESIGN_STRATEGY_SYSTEM.md`、`docs/USER_PERSONAS.md`、`openspec/changes/p4-research-interpretation-path/`

## 版本目标

v0.4.4 已经把首次体验和 `report latest` 收口成结果卡。v0.4.5 要继续往前推一层：不只是告诉用户“发生了什么”，而是告诉用户“为什么会这样”“在哪种市场状态下会这样”“什么时候可以继续观察，什么时候应该停手”。

这版不是实盘执行，也不是再加一个策略入口，而是把研究结论的解释能力补齐，让交易者能沿着同一条链路继续判断：

`结果是什么 -> 为什么这样 -> 在什么环境下这样 -> 是否值得进入只读观察`

## 版本承诺

1. 用户可以查看关键交易重放，理解入场、出场、失败和关键分歧点。
2. 用户可以按市场状态分段查看同一策略的表现，不再被单一汇总值掩盖差异。
3. 用户可以先进入只读观察边界，再决定是否值得继续看。
4. 多品种配置必须对每个 symbol 单独做 smoke-test，不能只验证第一个品种。
5. 系统不会把观察结论包装成实盘建议，也不会把纸面观察说成收益证明。

## 产品边界

### In Scope

- 关键交易重放：只重放关键入场 / 出场 / 失败节点，不做全量逐 bar 回放。
- 市场状态分段评估：至少区分牛市、熊市、震荡市和高波动环境。
- 只读观察边界：先定义虚拟订单、延迟、滑点和人工闸门，再考虑后续执行层。
- 多品种逐 symbol smoke-test：每个 symbol 都要显式验证和显式报告。
- 报告语言必须面向交易判断，不要把工程实现细节放到第一屏。

### Out of Scope

- 实盘执行。
- 自动下单。
- 全量 tick 级回放。
- 完整订单簿撮合引擎。
- 多用户控制平面。
- 研究自动化之外的投机式参数搜索。
- 把只读观察态伪装成实盘。

## 用户流程

### CLI / 报告路径

建议保持当前主入口不变：`kronos report latest` 仍然是第一屏。v0.4.5 在此基础上补三类独立下钻能力：

1. 从最新报告跳到关键交易重放。
2. 从关键交易重放跳到市场状态分段。
3. 从分段结论跳到只读观察边界。

策略侧需要补齐逐 symbol smoke-test，确保“一个策略配置里每个品种都能单独验证”。

### Agent / Web 路径

Agent 和 Web 不应只告诉用户“通过 / 未通过”，还要告诉用户：

1. 这次为什么通过或没通过。
2. 哪个市场状态支持，哪个市场状态不支持。
3. 哪些品种已经单独验证，哪些品种还没过。
4. 是否适合进入只读观察，而不是直接进入更强执行层。

### 只读观察路径

只读观察不是“换个名字的实盘”，而是一个带边界的观察态：

1. 只读交易权限或等价的非交易权限。
2. 虚拟订单记录，不发真实订单。
3. 显式延迟和滑点假设。
4. 人工闸门，避免把观察态误切到执行态。

## 功能需求

### 需求 1：关键交易重放必须解释策略为什么这样表现

系统 MUST 提供关键交易重放视图，让用户能看到策略在关键节点上为什么进入、为什么退出、为什么失败。

#### 验收点

- 能按 run / 策略定位到关键交易。
- 每个关键交易都至少包含入场原因、出场原因、结果和相关市场背景。
- 系统不会只给“收益 / 回撤”而不给过程解释。

### 需求 2：市场状态分段评估必须独立给结论

系统 SHALL 把同一策略在不同市场状态下的表现分开呈现。

#### 验收点

- 牛市、熊市、震荡市、高波动环境至少能被分别识别。
- 每个分段都必须有自己的表现摘要和结论。
- 不允许把一个明显差异很大的分段悄悄并入总均值。

### 需求 3：只读观察边界必须先于执行层定义

系统 SHALL 在进入只读观察前明确虚拟订单、延迟、滑点和人工闸门，不得把观察态描述成真实执行。

#### 验收点

- 观察态必须显式说明是否只读。
- 观察态必须记录虚拟成交规则和假设。
- 观察态必须和实盘执行在文案和状态上严格分开。

### 需求 4：多品种 smoke-test 必须覆盖所有声明品种

系统 SHALL 对策略配置中声明的每个 symbol 单独做 smoke-test。

#### 验收点

- 每个 symbol 都有自己的通过 / 失败结果。
- 某个 symbol 失败不能被其他 symbol 的成功掩盖。
- smoke-test 必须显示是哪个 symbol、哪一步、什么原因没过。

### 需求 5：报告必须保持产品语言

系统 MUST 继续把技术产物翻译成交易语言，但不得把观察结论包装成实盘信号。

#### 验收点

- 用户能先看懂结果，再看细节。
- 产品语言里必须保留“为什么”“在哪个环境下”“还能不能继续观察”。
- 系统不得输出“已证明可实盘”这类超出证据的结论。

## 版本完成标准

v0.4.5 完成时，必须满足：

1. 用户可以从最新报告一路下钻到关键交易和市场状态分段。
2. 用户可以明确知道策略在什么环境下更可靠、在什么环境下更脆弱。
3. 用户可以看到只读观察边界，但不会被误导为已经进入实盘。
4. 多品种策略不会只验证首个 symbol。
5. 版本文档、OpenSpec、TODO、PROJECT_STATUS、ROADMAP 和产品设计文档口径一致。

## 测试与验证

至少需要覆盖：

- 单元测试：关键交易提取、市场状态分段、只读观察状态、逐 symbol smoke-test。
- 集成测试：`report replay`、`report regime`、`report observation`、逐 symbol 失败反馈、观察边界文案。
- 文档验证：版本需求、OpenSpec、TODO、PROJECT_STATUS、ROADMAP 同步。
