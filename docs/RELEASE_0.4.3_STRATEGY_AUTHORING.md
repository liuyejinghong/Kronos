# Kronos v0.4.3 版本需求：AI 辅助策略起草

> 状态：已交付
> 版本目标：0.4.3
> 约束来源：`TODO.md`、`docs/PROJECT_STATUS.md`、`docs/PRODUCT_DESIGN_STRATEGY_SYSTEM.md`、`openspec/changes/p4-strategy-authoring/`

## 版本目标

v0.4.3 只做一件事：把用户的自然语言策略想法，转成可验证的策略草案，并继续走现有的 `validate → smoke-test → register → report` 链路。

这版不是“生成任何策略代码”，而是“把一个想法变成当前系统能理解、能校验、能继续研究的策略资产”。

## 版本承诺

1. 用户可以用自然语言描述策略意图。
2. 系统可以判断该意图是否落在当前支持的模板范围内。
3. 系统可以输出一份人能读懂的策略概要。
4. 系统可以输出一份可编辑、可校验的 TOML 草案。
5. 系统不会编造盈利结论、历史表现或不支持的能力。
6. 系统不会绕过现有验证链路。

## 产品边界

### In Scope

- 自然语言到策略草案。
- R-breaker 及未来受支持模板的模板匹配。
- 缺失信息时的澄清问题。
- 草案生成后可直接进入 `strategy validate` / `strategy smoke-test` / `strategy register`。
- CLI、Agent console 和 Docker 路径都必须可复制、可解释。
- 结果必须保持产品语言，不输出工程黑话。

### Out of Scope

- 不做任意新策略引擎。
- 不做历史重放。
- 不做实时模拟盘。
- 不做实盘执行。
- 不做自动下单。
- 不做多租户或远程协作版本。
- 不做“看起来像成功”的伪策略生成。

## 用户流程

### CLI

用户输入自然语言意图，系统给出策略概要、澄清问题或拒绝原因、TOML 草案和下一步命令。

建议入口：

```bash
kronos strategy draft --prompt "我想做一个日内突破策略..."
```

首版默认使用确定性模板规则；用户显式传 `--use-ai` 时，才会尝试使用本地 DeepSeek 配置辅助解析。

### Agent Console

用户在 `kronos agent start` 中表达“我有一个想法”后，Agent 可以把该想法导向策略起草流程，而不是直接给出跑偏的结论。

### Docker

Docker 场景下，输出的路径和下一步命令必须直接可复制，不能把宿主机路径误导给容器用户。

## 功能需求

### 需求 1：自然语言输入必须先被结构化

系统 MUST 将自然语言策略意图转成结构化策略概要，再决定是否进入草案生成。

#### 验收点

- 能识别策略类型倾向。
- 能识别品种、周期、方向、持仓逻辑和风险约束。
- 不能识别的部分必须明确列为“未确定”，不能静默补全。

### 需求 2：草案生成必须基于当前支持模板

系统 SHALL 基于已支持的模板生成草案。v0.4.3 首版必须至少支持当前系统已有的策略配置契约，默认以 R-breaker 模板为起点。

#### 验收点

- 草案输出必须符合现有策略配置结构。
- 如果意图不在当前支持模板内，系统必须明确拒绝，而不是伪造一个新策略。
- 草案必须显式展示系统做了哪些默认假设。

### 需求 3：系统必须先给出人能读懂的策略概要

系统 SHALL 在写出 TOML 之前，先给出简明策略概要，包括意图、模板、品种、周期、关键参数、默认假设和未确定项。

#### 验收点

- 用户可以先确认再落盘。
- 概要中不得出现“已盈利”“已证明有效”等结论。

### 需求 4：草案必须进入现有验证链路

系统 SHALL 让生成的草案直接进入现有 `validate → smoke-test → register` 路径，而不是绕开既有策略配置流程。

#### 验收点

- `strategy validate` 可校验草案。
- `strategy smoke-test` 可在有数据时试算草案。
- `strategy register` 仍是明确的人工可见动作。

### 需求 5：系统必须能解释拒绝和澄清

系统 MUST 在信息不足或模板不支持时给出产品化解释，并指出下一步动作。

#### 验收点

- 缺少关键字段时能提出澄清问题。
- 模板不支持时能明确说明原因。
- 不能把“不能做”伪装成“已经完成”。

### 需求 6：可追溯性必须保留

系统 SHALL 保留 prompt 版本、模型、模板命中、草案输出路径和最终动作的可追溯记录。

#### 验收点

- CLI / Agent 输出可以回溯到当次输入。
- 后续用户可以知道草案是怎么来的。

## 版本完成标准

v0.4.3 完成时，必须满足：

1. 自然语言输入能稳定导向一个可解释的策略草案。
2. 不支持的意图会被清楚拒绝。
3. 草案可以进入现有校验和烟雾测试链路。
4. Docker 和本地路径提示一致可复制。
5. 没有引入新的执行层、模拟盘或实盘能力。
6. 版本文档、OpenSpec、TODO、PROJECT_STATUS 和 CHANGELOG 口径一致。

## 测试与验证

必须至少覆盖：

- 单元测试：输入结构化、模板选择、澄清、拒绝、草案生成。
- 集成测试：CLI `strategy draft`、`validate`、`smoke-test`、`register` 串联。
- Docker 验证：路径提示、下一步命令、无宿主机路径误导。
- 文档验证：版本需求、OpenSpec、TODO、PROJECT_STATUS 同步。
