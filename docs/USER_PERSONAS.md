# Kronos 用户画像

> 日期：2026-05-06  
> 版本：0.4.0  
> 用途：统一产品定位、功能优先级和文案边界。

## 一句话结论

Kronos 的核心用户不是完全小白，而是**已经在交易、已经有策略想法、但缺少完整量化研究工程能力的人**。

“小白用户”仍然重要，但他们主要用于检验安装、首次引导和错误提示是否足够清楚；不应把整个产品做成“零金融知识也能自动赚钱”的工具。Kronos 更适合定位为本地优先的策略研究助手：帮用户把交易想法变成可验证证据，而不是替用户承诺收益或代替交易判断。

## 用户成熟度分层

| 层级 | 用户类型 | 是否核心用户 | Kronos 应该怎么服务 |
|---|---|---|---|
| L0 | 完全小白：不懂 K 线、回撤、手续费、模拟盘 | 否 | 只保证 quickstart 和 Docker 不吓人；不把复杂功能为他们降维成误导性承诺 |
| L1 | 入门交易者：会看 BTC/ETH，知道多空和止损，但不会写代码 | 次核心 | 用交易语言解释报告，明确“不能下单/不能保证收益” |
| L2 | 有经验的主观/半自动交易者：有规则想法，想验证它是否靠谱 | 核心 | 用对话/TOML/报告帮他完成“想法 → 证据 → 是否继续观察” |
| L3 | 会 Python/pandas 的交易研究者：能读代码，但不想搭整套研究基础设施 | 核心 | 提供可追溯数据、PIT-safe 查询、验证门禁、可扩展策略入口 |
| L4 | 量化工程师/小团队：关心架构、可测试性、候选池和 Agent 流程 | 次核心 | 提供清晰模块边界、可插拔因子、实验账本和本地资产保护 |

## 画像 1：有经验的主观交易者

**典型背景**

- 交易 BTC/ETH/SOL 现货或合约 1-5 年。
- 熟悉 K 线、趋势、震荡、突破、止损、手续费。
- 不一定会 Python，也不愿意先学一套量化框架。

**他为什么会用 Kronos**

他有一套交易想法，例如“BTC 突破前一天高点后追多，震荡时少做”，但不知道历史上是否真的有效。他要的不是因子平台，而是一个可信回答：这套想法值得继续观察吗？

**他最关心的问题**

- 这个策略过去有没有明显优势？
- 它是在什么市场状态下赚钱，什么状态下亏钱？
- 换成 ETH/SOL 后还成立吗？
- 如果暂时不行，是参数问题、样本问题，还是逻辑本身不行？
- 下一步该调参数、换品种、重放关键交易，还是放弃？

**产品要求**

- 报告第一屏必须用交易语言，不用先解释 `mean_rank_ic`。
- quickstart 和 Agent 应该围绕“这个策略能不能继续”组织，而不是围绕模块名组织。
- 必须反复说明当前不会模拟盘或真实下单。
- 参数调整要给交易含义，例如“突破倍数越高，信号越少，可能减少假突破”。

**不该怎么服务**

- 不要让他先实现 `Factor` 协议。
- 不要把候选池、生命周期、IC/ICIR 作为第一屏主概念。
- 不要把 sample 数据上的结果包装成策略有效性结论。

## 画像 2：半自动策略尝试者

**典型背景**

- 会把 TradingView 指标、手工复盘、群里策略或自己的经验总结成规则。
- 可能不会写完整代码，但能说清楚入场、出场、过滤条件。
- 愿意不断调参，但需要系统告诉他调参是否有证据。

**他为什么会用 Kronos**

他需要把自然语言策略快速变成研究对象，并看到失败原因。对他来说，Kronos 的价值不是“生成一段代码”，而是把策略放进一条固定验证链路：配置、烟雾测试、回测、分市场状态评估、模拟盘观察。

**他最关心的问题**

- 系统有没有准确理解我的策略？
- 触发买卖点是否符合我的原意？
- 参数改动是让策略更稳，还是只是过拟合？
- 什么时候可以进入模拟盘观察？

**产品要求**

- AI 策略创建必须先输出人能读懂的策略摘要，再生成 TOML。
- 烟雾测试要展示“为什么触发/为什么不触发”，否则用户无法发现理解偏差。
- 每次注册策略前都要有闸门：配置校验、逻辑试算、风险边界说明。
- 历史重放对这个画像比复杂指标更重要。

**不该怎么服务**

- 不要让 LLM 直接生成并执行未确认代码。
- 不要把“烟雾测试通过”说成“策略通过验证”。
- 不要在没有回放证据时承诺“AI 已经理解你的策略”。

## 画像 3：会 Python 的交易研究者

**典型背景**

- 会 pandas / notebook / 简单回测。
- 可能自己写过脚本，但数据下载、清洗、避免未来函数、实验记录很乱。
- 看得懂 IC、walkforward、回撤、换手、手续费，但不想每次重造基础设施。

**他为什么会用 Kronos**

他需要一个本地优先的研究底座：数据可追溯、查询 PIT-safe、因子可注册、验证有门禁、结果能沉淀。对他来说，Agent 是加速器，不是唯一入口。

**他最关心的问题**

- 数据是否干净，是否有未来函数？
- 实验是否可复现？
- 策略失败原因能否结构化沉淀？
- 我能不能把自己的因子接进来？
- 工具会不会污染我的本地策略资产？

**产品要求**

- 文档必须清楚说明数据来源、时间范围、API Key 边界。
- 保留技术指标和 JSON artifact，不要只给自然语言摘要。
- 测试和 demo 必须隔离真实 `~/.kronos` 状态。
- CLI 和 Python API 都要稳定，Web 只是一个查看面。

**不该怎么服务**

- 不要隐藏所有技术细节。
- 不要为了“易懂”牺牲可追溯性和可复现性。
- 不要把本地文件、候选池和报告目录做成黑盒。

## 画像 4：小型量化团队负责人

**典型背景**

- 有多个策略想法或策略来源。
- 需要决定哪些策略继续研究、哪些淘汰、哪些进入模拟盘观察。
- 关心流程、审批、证据一致性和团队协作。

**他为什么会用 Kronos**

他需要一个“策略候选池 + 研究证据 + Agent 复盘”的本地控制台。Kronos 对他的价值是让策略研究像产品管线：每个候选都有来源、状态、证据、失败原因和下一步。

**他最关心的问题**

- 当前候选池里哪些最值得继续？
- 为什么这个策略没有晋升？
- Agent 的建议依据是什么？
- 哪些决策需要人工审批？
- 团队成员能否复现同一份结论？

**产品要求**

- Web 工作台要优先显示候选状态、关键证据、失败原因分组和下一步。
- Agent 输出必须有证据引用，不只是“建议继续/淘汰”。
- 报告要能给非工程决策者读懂，同时保留原始 artifact。
- 需要明确模拟盘和实盘的人为闸门。

**不该怎么服务**

- 不要把 Agent 设计成自动交易员。
- 不要让候选状态只存在内存里。
- 不要让文档和真实版本能力分叉。

## 画像 5：量化工程接手者

**典型背景**

- 需要维护 Kronos 或把它接入已有研究/交易系统。
- 关心模块边界、测试覆盖、配置兼容、Docker 和本地开发体验。
- 不一定是最终策略使用者，但会决定项目能不能长期推进。

**他为什么会用 Kronos**

他看重的是工程底座：数据层、因子层、研究层、Agent 层和 Web 层能否分清责任。对他来说，用户体验问题也很重要，因为错误的产品承诺会反过来制造技术债。

**他最关心的问题**

- 版本事实源是否统一？
- 当前功能到底交付到哪一步？
- 新增策略类型应接在哪个边界？
- 测试是否会碰用户真实数据？
- Docker 是否能从干净环境启动？

**产品要求**

- `PROJECT_STATUS.md`、`TODO.md`、`CHANGELOG.md`、主设计文档必须同步。
- 每次新增产品能力都要有验收命令和失败路径。
- 文档要区分当前能力、目标体验和延期能力。
- 本地优先资产必须有测试隔离和路径边界。

**不该怎么服务**

- 不要只写愿景文档，不写当前可验证事实。
- 不要为一个 demo 快速硬编码会破坏后续策略系统的路径。

## 画像 6：安装小白 / Docker 首次用户

**典型背景**

- 可能是交易者，也可能是评测者。
- 不熟悉 Python 虚拟环境、Docker volume、Node/Web 前端。
- 只想知道“我跑起来没有，结果在哪，下一步做什么”。

**他是不是核心用户**

不是核心业务用户，但他是非常重要的**入口压力测试用户**。如果他跑不通 quickstart，Kronos 的传播和试用会失败；但如果为了他把所有研究概念都抹平，Kronos 会失去专业可信度。

**产品要求**

- `docker compose up` 要有明确进度、结果路径和下一步命令。
- 错误提示要告诉他“是网络、依赖、数据、还是配置问题”。
- 不要把 Docker 内部路径、volume 名称、构建层日志当成产品说明。
- quickstart 完成后必须能用一个命令读报告。

**不该怎么服务**

- 不要承诺“不懂交易也能用 Kronos 自动赚钱”。
- 不要让 Docker 用户按宿主机 Node/npm 指令操作。
- 不要把安装成功当成产品成功；成功标准是看到一份能读懂的策略报告。

## 产品优先级含义

### 当前主用户

v0.4.x 的主用户应锁定在 L2 和 L3：

- L2 有经验的交易者：需要把想法变成证据。
- L3 会 Python 的交易研究者：需要可靠研究底座和可扩展入口。

L1 和 L0 只决定 onboarding 的清晰度，不决定研究功能的深度。L4 决定长期架构质量，但不应让产品第一屏变成工程控制台。

### 功能排序

| 优先级 | 功能方向 | 服务谁 | 原因 |
|---|---|---|---|
| P0 | 报告交易语言、最新报告、Docker 主路径 | L1/L2/L6 | 保证首次体验能形成判断 |
| P0 | TOML 配置、烟雾测试、注册闸门 | L2/L3 | 让策略从样例变成用户资产 |
| P1 | AI 自然语言策略创建 | L2 | 降低从想法到配置的门槛 |
| P1 | 历史重放 / 关键交易回放 | L2/L3 | 让用户验证系统是否理解策略 |
| P1 | 市场状态分段评估 | L2/L3/L4 | 解释策略什么时候有效/失效 |
| P2 | 实时模拟盘 | L2/L4 | 需要更高信任和运行时边界 |
| P3 | 实盘执行 | L4+高阶用户 | 必须等待研究、模拟盘和审批链路稳定 |

### 文案原则

- 面向交易者时，说“策略想法、买卖点、回撤、手续费、是否值得继续观察”。
- 面向研究者时，说“数据快照、PIT-safe、walkforward、样本外稳定性、artifact”。
- 面向工程接手者时，说“模块边界、测试隔离、版本事实源、可复现命令”。
- 面向小白时，只简化安装和下一步，不简化风险边界。

## 对当前路线的判断

Kronos v0.4.0 的策略配置入口是正确方向，但它还不是“普通小白可以无脑使用”的产品。下一步不应先追求更多策略数量，而应围绕 L2/L3 用户补齐三件事：

1. **AI 创建策略**：让用户用自然语言表达想法，但输出必须落到 TOML 并经过 smoke-test。
2. **关键交易回放**：让用户看到系统是否按他的交易意图触发买卖。
3. **市场状态解释**：让用户知道策略失败是因为市场环境、参数、样本还是逻辑本身。

如果这三件事做好，Kronos 才会从“本地研究原型”变成“交易者愿意持续使用的策略研究 Agent”。
