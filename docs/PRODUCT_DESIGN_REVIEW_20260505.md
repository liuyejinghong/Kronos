## 一、产品经理视角

### 1.1 价值主张不够锋利

当前核心原则写的是"Kronos 是一个策略研究助手"。但交易者的真实问题是：

> "我有个策略想法，怎么知道它能不能赚钱？"

产品描述应该直接回答这个问题，而不是用"助手"这种模糊词。建议改为：

> Kronos 让你在 10 分钟内把一个策略想法变成可验证的结果。
> 不需要写代码，不需要搭回测框架，不需要连交易所。

### 1.2 缺少竞品参照

文档没回答"为什么不用 TradingView / Freqtrade / 自己写 Python"。Kronos 的差异化在哪？

| 竞品 | 优势 | Kronos 的差异化 |
|------|------|----------------|
| TradingView | 图表+PineScript | **AI 自然语言 → 策略代码**，不需要学 PineScript |
| Freqtrade | 开源回测+实盘 | **对话式交互**，Agent 主动引导，不是命令行 |
| 自己写 Python | 完全自主 | **10 分钟到手结果**，不用搭 infra |

这个对照表应该放在产品设计首页。

### 1.3 版本边界模糊

文档从 v0.3 写到 v0.5+，但没有标注哪些是**当前可用的**、哪些是**规划中的**。读者会认为所有功能都已实现——然后失望。

建议每个功能标注 `[v0.3]` / `[v0.4]` / `[v0.5]` 或使用 ✅ 🔄 ⏳ 标记。

### 1.4 R-breaker 作为 demo 策略的风险

如果 R-breaker 回测结果是负的，用户的第一印象是"这个工具不行"。Demo 策略必须是：
- **参数宽泛，在不同市场条件下都有过得去的结果**（不能只在牛市赚钱）
- **有明确的解释说明为什么亏钱是正常的**（"这是一个教学策略，帮你理解系统怎么用，不代表可以实盘"）

### 1.5 缺少数据依赖说明

整个策略系统建立在"有数据"的前提上。但：
- 数据从哪来？（Binance REST API）
- 需要什么权限？（不需要 API Key 也能拉 K 线）
- 能拉多少？（单次 1500 根，可以翻页）
- 离线能跑吗？（可以，`kronos quickstart` 生成 sample 数据）

这个数据依赖链条应该在一开始就说明清楚。

## 二、交易者用户视角

### 2.1 "R-breaker" 是什么，为什么要关心

一个普通 crypto 交易者看到 R-breaker 的第一反应：

> "这什么？没听过。关我什么事。"

**改进**：不要用策略名做入口，用**交易风格**做入口：

```
你想做什么类型的策略？

  [1] 顺势而为 — 在趋势确认后入场（适合 BTC/ETH 大周期）
  [2] 日内短线 — 当天进出，吃小波动（适合山寨币）
  [3] 我有一个具体的想法 — 描述你的交易逻辑
```

R-breaker 藏在选项 2 后面，作为一个**可以选择体验的例子**，而不是主入口。

### 2.2 TOML 配置是程序员思维

策略 TOML 里写 `condition = "close_pct_change(10) > 0.02"` — 这是代码，不是对话。

用户想说的是："过去 10 根 K 线涨了 2% 就买"。而不是写表达式。

**改进方案**：TOML 是**LLM 生成后的存储格式**，用户不应该直接编辑 TOML。用户通过对话定义策略，LLM 写入 TOML。如果要编辑，也是通过对话："把入场条件改成最近 5 根 K 线涨 1.5% 就买"。

### 2.3 回测结果没有上下文

当前设计里的回测输出：

```
· 总收益率: +3.2%
· 胜率: 52%
· 最大回撤: -4.1%
· 夏普比率: 0.68
```

这些数字对专业量化交易员有意义。但对一个普通交易者，他看到这些会问：

> "+3.2% 是好还是差？跟拿着不动比呢？这是 30 天的结果，如果是熊市 30 天呢？"

**改进**：结果必须带**对比基准**：

```
📊 R-breaker 近 30 天在 BTC 上的表现:

  策略收益: +3.2%
  同期持有 BTC: +1.8%（你的策略跑赢了持有）
  最大回撤: -4.1%（BTC 同期最大回撤 -6.3%）
  
  💡 你的策略在上升趋势中表现比持有好，但震荡时会亏。
     要不要看看在不同市场状态下的表现？
```

### 2.4 AI 策略创建缺少验证步骤

用户用自然语言描述策略 → LLM 生成 TOML → 用户确认 → 注册。

但这个流程漏了最关键的一步：**LLM 生成的代码对吗？**

如果 LLM 把"涨 2% 做多"理解成"跌 2% 做多"，用户确认时看不出来，跑出来的结果是错的。用户会以为策略不行，实际上代码有问题。

**改进**：确认后加一个**快速烟雾测试**：

```
Agent: 我为你的策略跑了 10 天的快速验证。这是其中 3 笔交易的逻辑:

  3月15日 14:00 — 过去10根K线涨了2.3% → 触发做多 ✅ 符合预期
  3月18日 09:00 — 过去10根K线跌了1.2% → 不触发 ✅ 符合预期  
  3月22日 20:00 — 做多持仓满5根K线 → 平仓 ✅ 符合预期

——逻辑确认正确，要跑完整回测吗？
```

### 2.5 执行链路跳过了"模拟盘离线模式"

设计里模拟盘需要 Binance API Key。但新用户可能：
- 没有 Binance 账户
- 不想给 API Key
- 只想看"如果我在过去 30 天用这个策略，每天会怎么交易"

**改进**：增加一个**历史重放模式**（不需要 API Key）：

```
Agent: 要体验执行流程，有两种方式:

  [1] 历史重放 — 用过去 30 天数据模拟实时交易，
      你可以看到每一天 Agent 会做什么（不需要 API Key）
  [2] 实时模拟盘 — 连接 Binance 实时行情（需要只读 API Key）

> 1

Agent: 好的。我会像在真实交易中一样，每分钟处理一根 K 线。
       你会看到每秒的信号、订单、持仓变化——只是发生在过去的数据上。

       开始重放 3月15日…
       [09:00] BTC 68720 → 无信号
       [09:01] BTC 68680 → 无信号
       [09:15] BTC 69100 → R-breaker 做多信号 → 虚拟买入 0.02 BTC
       [13:45] BTC 69450 → 达到止盈 → 虚拟卖出 0.02 BTC
       ...
```

### 2.6 缺少迭代闭环

设计里策略验证完 → 结果 → 然后呢？如果结果不好，用户怎么改进？

当前只有一个"调整参数"的入口。但策略改进不仅仅是调参数——可能需要换个时间框架、加个过滤条件、换个品种。

**改进**：每个验证结果后面带**可执行的迭代建议**：

```
💡 根据这次回测，以下是具体可做的改进:

  [1] 加一个震荡过滤器 — 只在 ADX > 25 时开仓，减少假信号
  [2] 调整止盈倍数 — 从 1.5x 提高到 2x，让利润跑更远
  [3] 缩小时间框架 — 从 15m 改成 5m，更快响应
  [4] AI 帮我分析改进方向
```

## 三、需要补全的内容

| # | 缺失内容 | 影响 |
|---|----------|------|
| 1 | **数据故事** — 数据从哪来、怎么获取、需要什么权限 | 用户第一步就卡住 |
| 2 | **竞品对比** — 为什么不用 TradingView / Freqtrade | 用户不知道差异化 |
| 3 | **版本标注** — 哪些功能现在可用，哪些在规划 | 期望管理 |
| 4 | **失败处理** — LLM 生成策略出错后怎么修正 | 用户信心 |
| 5 | **烟雾测试** — 策略代码正确性快速验证 | 防止策略逻辑错误 |
| 6 | **历史重放模式** — 不依赖 API Key 的执行体验 | 降低体验门槛 |
| 7 | **对比基准** — 回测结果 vs 持有 vs 市场指数 | 结果可理解 |
| 8 | **迭代闭环** — 验证完了之后怎么改进 | 产品粘性 |
| 9 | **策略复杂度上限** — LLM 能处理什么、不能处理什么 | 预期管理 |

## 四、修订建议

1. **首页加价值主张**：10 分钟从想法到验证结果
2. **入口改为交易风格**：顺势/日内/自定义，而非 R-breaker
3. **TOML 是内部格式**：用户不直接编辑，通过对话创建和修改
4. **回测结果带基准**：vs 持有、vs 市场指数
5. **AI 策略加烟雾测试**：确认逻辑正确再跑完整回测
6. **增加历史重放模式**：不需要 API Key 就能体验执行流程
7. **每个结果带迭代建议**：参数、过滤、品种、时间框架
8. **标注版本边界**：所有功能标 `[v0.3]` / `[v0.4]` / `[v0.5+]`
