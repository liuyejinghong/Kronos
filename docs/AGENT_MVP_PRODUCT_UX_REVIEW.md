# Kronos Agent MVP 产品体验复盘

日期：2026-05-01

复盘方式：以“小白量化交易员”身份打开本地 Web 工作台，通过浏览器实际点击和滚动使用，不站在开发者视角解释页面。

最新二次走查：

- 2026-05-01 使用 Computer Use 完整点击首页、报告、候选池、候选详情、补材料、模型配置、审批和时间线。
- 结论：第一批 UX 修复后，页面从“长页面研发验收台”变成了“有模块导航的工作台”，但仍未达到产品可用 MVP。
- 详细记录见 [AGENT_MVP_PRODUCT_WALKTHROUGH_AUDIT_20260501.md](AGENT_MVP_PRODUCT_WALKTHROUGH_AUDIT_20260501.md)。

复盘页面：

```text
http://127.0.0.1:3000/
```

## 一句话结论

当前 Web 工作台已经把 Agent 批次、候选池、时间线、模型配置和材料入口展示出来了，但它仍然更像 **研发验收页**，还不像一个普通量化交易员可以顺着完成任务的产品。

最大问题不是缺少解释文案，而是缺少清晰的产品任务流：用户打开后不知道“我现在该做什么、先看什么、能点哪里、这轮结论从哪来、下一步如何推进”。

因此，当前状态不能按“产品可用 MVP”验收。下一步应先做 **产品 UX 修复批次**，再继续扩展更深的量化能力。

## 体验假设

体验者不是项目开发者，只是一个普通量化交易员。

他打开 Kronos 的合理预期是：

1. 先知道 Kronos 当前在研究什么。
2. 能读懂 Agent 给出的结论和证据。
3. 能看到候选策略或因子在哪里。
4. 能判断哪些候选值得继续验证。
5. 能知道自己是否需要配置模型、导入材料或审批下一步。
6. 如果有报告，应该能直接阅读报告，而不是看到本地文件路径。

## 实际体验记录

### 1. 打开首页

第一屏能看到项目名、Agent 状态、候选资产数量、人工闸口、模型配置和本地 API。

问题是：它没有形成一个自然的操作起点。

用户看到“待命”后会理解系统当前没有在跑，但不知道下一步应该点哪里。页面只有“刷新”按钮，刷新不是下一步研究动作，也不是产品主任务。

产品判断：

- “待命”只是机器状态，不是用户任务状态。
- 第一屏缺少主操作，例如“阅读本轮报告”“查看候选改造建议”“配置模型”“导入研究材料”“开始下一轮研究”。
- 如果当前阶段不能开始下一轮研究，也应该明确展示为什么不能开始，以及当前最推荐的下一步。

### 2. 查看报告与证据

页面展示了“报告与证据”，但内容主要是 `run report`、`json`、`md` 和一串文件路径。

我尝试点击报告路径，没有出现报告阅读器、抽屉、弹窗或新页面。

产品判断：

- 文件路径对开发者有意义，对量化交易员没有意义。
- 报告应该是可读产品资产，而不是仓库索引。
- 如果是 Markdown 报告，页面应该内嵌渲染摘要和正文。
- JSON 可以保留给调试，但不应该作为产品主展示。

预期形态：

- 报告卡片显示“本轮研究报告”“Agent 研究计划”“Agent 决策说明”。
- 点击后展开右侧抽屉或进入报告页。
- 报告页包含结论、证据、反方意见、风险、下一步动作和原始附件。
- 文件路径可以折叠在“技术详情”里。

### 3. 进入候选池

点击左侧“候选池”后，页面跳到候选策略表格。

这一步证明候选资产确实存在，但作为产品体验仍然不够清晰：

- 候选池不是独立页面，只是长页面中的锚点。
- 表格默认展示很多偏工程字段，例如实现标识、英文研究族群、迁移分。
- 用户能看到排名，但不知道“为什么这个排第一”“下一步该研究哪个”“哪个已经被 Agent 选中”。
- 行可以点击，但页面没有明显提示“点击候选查看详情”。

产品判断：

- 候选池应该是核心页面，不应该藏在一页式长页面中。
- 候选行应该回答交易员关心的问题：它是什么策略资产、来自哪里、为什么值得迁移、当前证据够不够、下一步应该做什么。
- 当前表格像资产清单，不像研究决策工作台。

### 4. 点击候选项

点击“趋势内回踩入场”后，候选行被选中，向下滚动能看到候选详情。

详情页能展示生命周期、迁移分、来源和 Agent 分歧，但主要内容还是状态说明：

- “等待 Agent 选择是否进入下一轮研究。”
- “暂未绑定专项验证证据。”
- “仍需要 crypto 市场专项证据确认。”

产品判断：

- 候选详情已经有结构，但缺少行动闭环。
- 如果等待 Agent 选择，用户应该看到“让 Agent 研究这个候选”的入口，或看到为什么当前不能启动。
- 如果缺少证据，应该有“需要补什么证据”的清单。
- 如果这是本轮 Agent 已经提到的候选，应在候选详情中明确标注“本轮 Agent 结论关联此候选”。

### 5. 查看候选分布图

候选池下方有“候选分布”柱状图，展示不同研究族群的数量。

问题是：用户不知道这个图要帮助他做什么决策。

产品判断：

- 图表不是问题，问题是图表没有决策含义。
- “trend momentum 5 个、volatility path 4 个”只说明数量，不说明哪个更值得投研资源。
- 如果图表保留，应该转成研究资源视角，例如“高优先级候选主要集中在哪类”“哪个族群证据最弱”“哪个族群值得下一轮优先验证”。
- 如果无法提供决策含义，MVP 首页不应该放这个图。

### 6. 查看时间线

点击“时间线”后，页面跳到 Agent 时间线区域，同时右侧还能看到操作台。

时间线能看到工具执行事件，但对普通用户仍偏底层：

- “agent_propose”“agent_conclude”是内部动作名。
- 用户能猜到系统执行过步骤，但不知道这些步骤对研究结论的贡献。
- 时间线中也出现报告路径，仍然不可直接阅读。

产品判断：

- 时间线应该服务于“Agent 正在做什么”和“它为什么这么做”。
- 内部工具名可以保留，但不应该是主表达。
- 每个事件应该能展开为人类可读结果，例如“提出研究计划”“完成候选选择”“生成结论报告”。

### 7. 查看操作台和模型配置

操作台包含模型、材料、审批三个标签。

模型配置显示 DeepSeek API Key 尚未配置；但首页同时已经展示 Agent 结论，角色表里也显示多个 Agent 角色启用。

这是当前最影响信任感的问题之一。

产品判断：

- 如果模型未配置，却展示 Agent 结论，用户会自然怀疑“这些 Agent 结论从哪来的”。
- 当前真实情况更像“本轮是确定性验收批次或历史批次，不依赖 DeepSeek 实时调用”。
- 页面必须明确区分：
  - 本轮结论来源：确定性工具 / 历史报告 / LLM Agent 实时分析。
  - 模型配置状态：DeepSeek 尚未配置，所以不能启动新的 LLM 多角色分析。
  - 角色启用状态：角色模板可用，不等于模型已可调用。

建议把“未配置”改成更完整的状态卡：

```text
DeepSeek 未配置
当前展示的是历史/确定性验收批次，不是 DeepSeek 实时生成。
配置模型后，可启动多角色 Agent 研究。
```

### 8. 查看材料入口

材料入口可以填写标题、材料类型、候选绑定、标签和内容。

这是比较接近真实产品功能的一块，但仍有两个体验问题：

- 它藏在长页面下方，用户很难从第一屏知道“我可以投喂材料”。
- 导入按钮禁用是合理的，但没有清楚提示最低必填项。

产品判断：

- 材料导入应该是首页主动作之一，尤其当前 Agent 待命时。
- 如果“有材料才继续跑”，那待命状态下最合理的行动就是“导入材料”或“选择候选开始研究”。

### 9. 查看审批入口

审批页显示当前没有待审批项，并列出 Prompt 启用、候选实现、模拟盘准入、组合准入、实盘申请。

用户能理解“现在没有要审批的事”，但不清楚这些按钮/标签未来如何触发。

产品判断：

- 审批中心方向是对的，但当前只是状态说明。
- 需要把审批和生命周期绑定起来：候选何时会进入审批、审批后发生什么、未审批时系统会停在哪里。
- 这里尤其要和用户关心的“实盘准入许可”和“资金大小”形成明确边界。

## 关键问题分级

### P0：没有明确的主任务流

用户打开后不知道下一步做什么。

这不是加一段说明文案能解决的，而是需要把页面改成任务驱动：

- 如果 Agent 待命：给出“导入材料”“选择候选研究”“配置模型”的主动作。
- 如果本轮已完成：给出“阅读报告”“查看推荐候选”“决定下一轮研究方向”的主动作。
- 如果模型未配置：明确哪些功能可用、哪些功能不可用。

### P0：报告不可阅读

当前把报告当文件路径展示，产品上不可接受。

报告是 Agent 产品的核心交付物，必须在 Web 里可读、可展开、可追溯。

### P0：模型状态和 Agent 结论存在信任冲突

页面同时出现“模型未配置”和“Agent 结论/多角色启用”，会让用户怀疑系统在伪造 Agent 结论。

必须补“结论来源”和“模型可用状态”的产品表达。

### P1：一页式长页面不适合这个产品

Kronos 不是展示型官网，而是研究工作台。

当前页面把首页、候选池、详情、时间线、操作台、配置、材料、审批都塞在一页里，导致信息架构不清晰。

应改成应用式结构：

- 首页 / 今日工作台
- 研究批次详情
- 候选池
- 报告与证据
- 操作台
- 设置

### P1：候选池展示像数据表，不像研究决策

候选池目前只是资产列表。

交易员更关心：

- 这个候选为什么重要？
- 当前证据够不够？
- Agent 为什么推荐或不推荐？
- 下一步需要做什么实验？
- 是否可能进入模拟盘或实盘准入？

### P1：图表没有决策含义

候选分布图当前只展示分类数量，无法指导用户行动。

MVP 应减少装饰性图表，优先展示“下一步研究建议”和“证据缺口”。

### P2：文案偏解释系统，而不是服务任务

页面上很多话是在解释 Kronos 是什么，而不是帮助用户完成当前任务。

更好的产品表达不是增加更多解释，而是把信息组织成可行动的模块。

## 建议的信息架构

### 首页：今日工作台

首页只回答五个问题：

1. Kronos 当前在做什么？
2. 本轮结论是什么？
3. 我是否需要操作？
4. 当前最重要的候选是什么？
5. 有哪些证据和风险？

首页主动作：

- 阅读本轮报告
- 查看推荐候选
- 导入新材料
- 配置模型
- 开始下一轮研究（如果暂不可用则禁用并说明原因）

### 研究批次详情

用于展示某一轮 Agent 研究：

- 研究目标
- 参与角色
- 结论
- 支持理由
- 反对理由
- 工具执行记录
- 证据报告
- 下一步建议
- 是否需要审批

### 候选池

候选池应是独立页面：

- 按“推荐研究、证据不足、观察中、淘汰、可进入模拟盘”等状态分组。
- 支持按来源、研究族群、生命周期、迁移分筛选。
- 点击候选打开详情抽屉或详情页。
- 每个候选明确展示“下一步动作”。

### 报告与证据

报告应该成为一级对象：

- 报告标题
- 生成时间
- 来源批次
- 关联候选
- 摘要结论
- 全文阅读
- 原始文件和 JSON 附件折叠展示

### 操作台

操作台保留三类能力：

- 模型配置
- 材料导入
- 人工审批

但操作台不应该替代首页主任务。首页应把最重要动作直接露出。

## 下一批开发建议

建议把下一批命名为：

```text
Product UX Repair Batch
```

优先任务：

1. 拆分一页式布局，形成首页、候选池、报告、操作台等明确视图。
2. 首页新增主动作区，根据 Agent 状态显示推荐动作。
3. 报告与证据改为可阅读报告卡片和报告阅读器。
4. 增加“结论来源”标识，解决模型未配置但已有结论的信任冲突。
5. 候选池改成决策型列表，突出推荐原因、证据缺口和下一步动作。
6. 候选详情绑定本轮 Agent 结论和相关证据。
7. 时间线事件改成人类可读表达，内部工具名降级到技术详情。
8. 删除或重做没有决策含义的候选分布图。

## 修复落地记录

2026-05-01 已完成 Product UX Repair Batch 第一批：

- Web 从单页长滚动改为左侧模块导航：首页、候选池、报告、时间线、操作台。
- 首页改成任务入口，直接给出“阅读报告、查看候选、导入材料、配置模型、下一轮研究”的当前动作。
- 报告不再把文件路径作为主要内容；默认展示可读摘要和正文，原始路径与产物路径折叠到“技术路径”。
- 首页明确标出“本轮结论来源”，说明当前批次来自本地确定性验收和历史报告读取，不是 DeepSeek 实时生成。
- 候选池改为决策表：当前判断、证据缺口、迁移分和详情入口成为核心字段。
- 候选详情补充“补材料”动作，避免只显示状态而没有下一步。
- 时间线事件改成人类可读标题，工具路径降级到技术详情。
- 模块切换会自动回到页面顶部，避免从报告页切到候选池时标题不可见。

这批修复解决的是“能不能像产品一样看懂和操作”的第一层问题。下一批仍需要继续做真正的 Agent 启动入口、候选详情证据挂载、图表和报告质量增强。

## 产品验收标准

下一版 Web 才能进入产品验收，需要满足：

- 普通量化交易员打开 5 秒内知道当前状态和下一步动作。
- 用户可以在 Web 内阅读本轮报告，不需要复制文件路径。
- 用户可以 1 次点击进入候选池，并理解候选排序和推荐理由。
- 用户可以看出每个候选下一步是研究、补证据、观察、淘汰还是审批。
- 页面明确说明本轮结论来自历史/确定性工具还是 LLM Agent。
- 模型未配置时，不出现“看起来已经由模型生成”的误导性状态。
- 首页不再依赖长页面滚动承载所有功能。
- 每个核心模块都回答一个交易员的决策问题，而不是只展示工程状态。

## 当前结论

当前 MVP 的后端链路和数据展示已经有基础，但 Web 产品体验还没有过线。

我建议先暂停继续堆新量化功能，优先修正产品信息架构和核心任务流。否则即使后端继续增加能力，用户仍然会觉得它“不像产品”，也不知道怎么验收和使用。
