# Kronos Web 二次产品走查记录

日期：2026-05-01

走查方式：按用户要求，使用 Computer Use 操作 Chrome，假设自己是第一次打开 Kronos 的产品经理 / 入门量化交易员。走查目标不是解释实现，而是验证“我能不能顺着页面完成一次 Agent 研究使用流程”。

走查地址：

- `http://127.0.0.1:3000/`
- `http://localhost:3000/`

## 总结论

当前 Web 仍不能按“产品可用 MVP”验收。

用户给 10/100 的评价是成立的。原因不是页面完全没有内容，而是核心任务流没有打通：用户打开后能看到历史批次、候选表、报告和配置页，但还不能自然完成“给 Agent 材料 -> 让 Agent 研究 -> 查看推理过程 -> 读报告 -> 决定下一轮动作”的闭环。

第一批 UX 修复让页面从“长页面堆信息”变成了“有模块导航的工作台”，但它仍然更像研发验收台，不像一个真正可用的 Agent 产品。

## 当前能做什么

- 可以在 `localhost:3000` 打开工作台。
- 可以查看上一轮历史 / 确定性验收批次。
- 可以查看 12 个候选策略 / 因子。
- 可以点开候选详情，看到迁移分、生命周期、证据缺口和来源。
- 可以进入模型配置页，看到 DeepSeek 尚未配置。
- 可以导入一条研究材料到材料池。
- 可以查看一条 Agent 时间线。

## 当前不能做什么

- 不能启动下一轮 Agent 研究。
- 不能看到真正的多 Agent 实时研究过程。
- 不能让刚导入的材料立刻触发 Agent 继续分析。
- 不能在 Web 内展开阅读技术附件，只能看到文件路径。
- 不能把报告理解为一份真正的量化研究报告。
- 不能从候选详情直接进入“为这个候选生成实验计划 / 发起验证”。
- 不能从审批中心理解“何时会需要我批准、批准后会发生什么”。

## P0 阻断问题

### P0-01：用户按 `127.0.0.1:3000` 打开会进入异常状态

复现路径：

1. 在 Chrome 打开 `http://127.0.0.1:3000/`。
2. 页面可见，但状态显示 `API 待连接`、候选为 `0`。
3. 实际接口通过本地请求可以返回候选数据。
4. 切换到 `http://localhost:3000/` 后页面恢复为 `API 已连接`、候选为 `12`。

影响：

用户只会认为产品坏了，不会知道这是本地开发域名导致。更严重的是页面不是直接报错，而是展示一个半可用状态，容易误导用户做错误判断。

期望：

- `127.0.0.1` 和 `localhost` 都应该可用；或自动重定向到可用地址。
- 如果 API 失败，页面必须给出明确错误原因和恢复动作。
- 不应该出现“接口实际可用，但页面显示空数据”的状态。

### P0-02：核心 Agent 闭环没有入口

复现路径：

1. 打开首页。
2. 看到 `Agent 状态：待命`。
3. 首页主动作里有 `开始下一轮研究`，但按钮是 `暂不可用`。
4. 导入材料后回到首页，仍然没有可执行的下一轮研究入口。

影响：

Kronos 的定位是 Agent，不是静态看板。用户最关心的是“我给你材料之后，你能不能开始研究并产出下一步”。当前最核心的动作被禁用，产品就还没有跑起来。

期望：

- 待命状态下应提供明确动作：选择候选、导入材料、启动研究。
- 如果不能启动，必须说明阻断原因，例如“缺模型配置 / 缺候选 / 缺材料 / 该入口未实现”。
- 导入材料后应出现“用这条材料启动研究”或“加入下一轮研究队列”。

### P0-03：报告仍不是量化研究报告

复现路径：

1. 首页点击 `打开报告`。
2. 报告标题是 `Kronos Agent 研究报告：20260430-agent-mvp-delivery-v1`。
3. 正文包含 `MVP 交付验收`、`Run ID`、`agent-cycle`、`任务数量`。

影响：

普通量化用户想看的是研究结论、证据、风险、实验计划、是否值得继续验证。当前报告像研发验收记录，不像可读的研究报告。

期望：

- 报告标题和结构应围绕研究对象，而不是交付批次。
- 首屏必须回答：研究了什么、结论是什么、证据够不够、下一步做什么。
- `Run ID`、内部任务名、文件路径应放入技术详情折叠区。

### P0-04：时间线附件仍只是路径，不是可读证据

复现路径：

1. 打开 `时间线`。
2. 展开 `agent_plan_report`。
3. 页面只显示 `reports/research/.../agent_research_plan.md` 路径。

影响：

用户点击报告附件，是为了阅读内容，不是为了看到仓库路径。路径展示会让用户再次回到开发者视角，无法完成产品内闭环。

期望：

- 点击附件应打开报告阅读器、侧边抽屉或详情弹窗。
- 默认显示摘要，提供“查看原文”和“查看技术路径”。
- JSON 附件可以保留，但不应作为默认阅读入口。

### P0-05：模型未配置与 Agent 结论并存，仍然伤害信任

复现路径：

1. 首页显示 `模型配置：未配置`。
2. 同时首页展示 `Agent 当前结论`、`支持理由`、`反对理由`。
3. 操作台模型页显示多角色 Prompt 模板已启用。

影响：

用户自然会问：“没有配置 DeepSeek，这些 Agent 结论从哪里来的？”虽然页面有一条“本轮非 DeepSeek 实时生成”，但信息仍然不够强，且角色表容易制造误解。

期望：

- 在结论旁边明确标注来源：历史批次 / 确定性工具 / LLM 实时生成。
- 角色表应该写成“Prompt 模板可用，尚未接入模型”，避免让用户以为 Agent 已经在跑。
- 新研究入口应和模型状态绑定。

## P1 重要体验问题

### P1-01：所有模块仍共享 `/`，没有真正页面路由

候选池、报告、时间线、操作台都在同一个地址 `localhost:3000/` 下切换。用户无法复制链接给别人，也无法刷新后保持当前位置。

期望：至少有 `/candidates`、`/reports/:run_id`、`/timeline/:run_id`、`/operations` 这类可恢复路径。

### P1-02：候选池仍像资产清单，不像研究决策台

候选表比上一版清晰，但仍有明显问题：

- `候选资产`这个叫法容易误解，实际展示的是策略 / 因子 / 研究候选。
- 研究族群仍然是英文标签，例如 `trend momentum`、`legacy mine`。
- 多数候选的证据缺口高度模板化，缺少个体差异。
- 没有筛选、搜索、状态分组。
- 本轮报告提到的候选没有被高亮为“本轮关联候选”。

期望：候选池先展示“本轮推荐研究对象”和“为什么”，再展示完整列表。

### P1-03：候选详情的“补材料”没有形成连续动作

复现路径：

1. 在候选池选中 `趋势内回踩入场`。
2. 向下滚动到候选详情。
3. 点击 `补材料`。
4. 页面跳到操作台材料表单，但没有稳定保留“我正在给哪个候选补材料”的上下文，需要重新确认候选绑定。

影响：

用户从候选详情来的意图是“给这个候选补证据”，系统应该自动保持上下文。当前动作割裂，用户要重新选择候选，容易补错对象。

期望：

- 从候选详情进入材料表单时自动绑定该候选。
- 表单顶部显示“正在给：趋势内回踩入场 补材料”。
- 导入后返回候选详情，并提示“材料已加入该候选证据池，下一步可启动研究”。

### P1-04：材料导入成功后没有下一步

本次走查实际导入了一条测试材料：

```text
UX 走查测试材料：趋势内回踩入场
```

页面提示 `已写入材料池`，但没有材料列表、没有跳转、没有“用这条材料启动研究”，也没有告诉用户 Agent 何时会读取它。

期望：

- 导入成功后显示材料记录。
- 告诉用户材料已关联到哪个候选。
- 提供下一步：启动研究、继续补材料、返回候选详情。

### P1-05：时间线是流水账，不是 Agent 思考过程

时间线能看到 `启动研究`、`开始执行工具`、`完成工具执行`，但每一步没有可读结果。用户无法理解 Agent 为什么得出结论。

期望：

- 每个事件显示“输入是什么、输出是什么、影响了什么结论”。
- 内部工具名可保留在技术详情。
- 时间线应支持按角色过滤，例如研究员、反方审查、风控审查、投委会裁决。

### P1-06：审批中心只有空状态，没有规则

审批页显示当前没有待审批项，并列出 Prompt 启用、候选实现、模拟盘准入、组合准入、实盘申请。

问题是用户不知道这些审批什么时候出现，也不知道每个审批会卡住哪一步。

期望：

- 显示审批规则：哪些动作需要人类批准。
- 显示生命周期位置：候选实现、模拟盘、组合、实盘分别在哪里停住。
- 和用户关心的“实盘准入许可”绑定。

## 逐页走查记录

### 首页 / 今日

可理解部分：

- 能看到系统叫 Kronos Agent。
- 能看到 API、候选数量、模型状态。
- 能看到本轮不是 DeepSeek 实时生成。

主要问题：

- 第一主动作仍不是“启动 / 继续 Agent 研究”。
- `开始下一轮研究`被禁用，是最大的产品阻断。
- 首页把“产品说明、验收批次、历史结论、候选推荐、配置状态”混在一起，任务层级仍不够清晰。

### 报告

可理解部分：

- 有可读报告页，不再只是路径列表。
- 能看到结论、建议动作、是否审批。

主要问题：

- 报告仍然是“交付验收报告”，不是“量化研究报告”。
- 结论和建议动作重复。
- 没有将报告关联回候选详情和下一步研究动作。

### 候选池

可理解部分：

- 能看到 12 个候选。
- 点击详情会更新下方候选详情。
- 迁移分和当前判断比旧版更容易看懂。

主要问题：

- 详情在页面下方，需要滚动；点击后没有明显“已打开详情”的空间变化。
- 没有筛选和分组。
- 证据缺口和下一步动作仍然缺乏具体实验指向。

### 候选详情

可理解部分：

- 生命周期结构是正确方向。
- 能看到迁移优先级、迁移分、来源、Agent 分歧。

主要问题：

- 大量状态是“等待 Agent 选择”，但没有让 Agent 选择的入口。
- “补材料”不够连续。
- 没有证据文件阅读入口。

### 操作台 / 材料

可理解部分：

- 表单能填写并导入。
- 可绑定候选。

主要问题：

- 导入后没有材料池列表。
- 没有告诉用户 Agent 何时读取。
- 没有把材料导入和下一轮研究连接起来。

### 操作台 / 模型

可理解部分：

- 能看到 DeepSeek API Key 未配置。
- 能看到多角色 Prompt 模板。

主要问题：

- Prompt 版本、角色定位仍偏工程。
- 没有“测试连接”或“配置后能解锁哪些功能”的明确说明。
- 不应该让用户误以为没模型也已有多 Agent 结论。

### 操作台 / 审批

可理解部分：

- 能知道当前没有待审批项。

主要问题：

- 没有审批规则和触发条件。
- 没有和候选生命周期形成闭环。

### 时间线

可理解部分：

- 能看到一次历史批次的事件顺序。

主要问题：

- 仍偏内部工具流水。
- 附件不能展开阅读。
- 无法看出每一步如何影响最终结论。

## 下一步修复优先级

### 第一优先级：把 Agent 主循环做成产品动作

目标不是继续美化页面，而是让用户能完成：

```text
选择候选 / 导入材料 -> 启动 Agent 研究 -> 观察 Agent 正在做什么 -> 阅读报告 -> 决定下一步
```

最小可验收版本：

- 首页有可点击的 `启动下一轮研究`。
- 未配置模型时清楚说明阻断。
- 有材料后能加入研究队列。
- 研究运行时有实时状态和事件。
- 完成后生成可读报告。

### 第二优先级：重做报告与证据阅读

报告要从“交付验收记录”改成“量化研究交付物”。

必须包含：

- 研究对象
- 研究假设
- 数据与证据
- 支持观点
- 反方观点
- 风险边界
- 下一步实验
- 是否需要审批

### 第三优先级：候选详情变成行动页

候选详情应有明确动作：

- 补材料
- 生成实验计划
- 启动回测 / 验证
- 请求反方审查
- 加入观察
- 申请模拟盘

当前只有“看状态”，还不是“推进研究”。

### 第四优先级：补页面路由和状态恢复

每个核心模块都应该有可恢复地址。刷新后不能丢失用户所在页面。

## 产品验收标准

下一轮修复后，应按以下任务验收：

1. 用户从空白浏览器打开正确地址，5 秒内知道 Kronos 当前在做什么。
2. 用户能判断为什么现在不能 / 能启动下一轮研究。
3. 用户能选择一个候选并为它补材料。
4. 补材料后，用户能看到材料被哪个候选接收。
5. 用户能启动一次研究或看到明确阻断原因。
6. 用户能看懂 Agent 时间线，不需要知道内部工具名。
7. 用户能在 Web 内阅读报告和证据，不需要看文件路径。
8. 用户能知道哪些动作需要人工审批，尤其是模拟盘、组合和实盘。

## 结论

当前应该暂停继续堆新模块，先修“Agent 产品闭环”。

如果下一步继续只优化候选表、报告卡片或布局，仍然会偏离用户预期。Kronos 的 MVP 不应该是一个能展示历史结果的看板，而应该是一个本地 Agent：有材料就研究，没材料就等待，研究中可观察，研究后给出可执行的下一步。
