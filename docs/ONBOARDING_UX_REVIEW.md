# Kronos 新用户 Onboarding 体验审查

> 审查时间：2026-05-04
> 审查方式：全新 Python venv 模拟新用户从零开始的完整路径

## 模拟过程

1. 复制项目到全新目录 `/tmp/kronos-fresh`
2. 创建全新 Python 3.12 venv
3. `pip install uv && uv sync --dev`
4. 按新用户直觉依次尝试：`--help` → `data status` → `pytest` → `agent --help`

---

## 发现的问题

### 🔴 1. 没有 README

**根因**：项目完全没有 README.md。新用户拿到代码后不知道这是什么、能做什么、怎么开始。

**影响**：任何人在 GitHub 上看到这个 repo 会直接关闭。即使是内部协作者，也只能靠阅读源码猜测入口。

### 🔴 2. `data sync` 对普通用户不可达

**根因**：数据管线依赖 Binance API + SOCKS 代理（中国大陆环境）。新用户执行 `kronos data sync` 会直接网络超时。

**影响**：整个研究管线建立在"有数据"的前提上。没有数据，因子计算、验证、Agent 全部不可用。新用户的第一步就被阻塞。

### 🟡 3. `agent run-once` 门槛过高

**根因**：Agent 命令要求 `--summary-json` 和 `--evidence-json` 作为必填参数，这两个文件是前置研究步骤的产物。

**影响**：即使用户有数据，也不知道怎么生成这两个 JSON。产品最有价值的功能对新手不可达。

### 🟡 4. config 路径脆弱

**根因**：所有命令默认 `--config configs/dev.toml`，这是相对于当前工作目录的相对路径。用户从其他目录运行时会报 `Config file not found`。

**影响**：用户必须 `cd` 到项目根目录才能执行任何命令。这在自动化脚本、cron 任务、或从其他目录调用时都会失败。

### 🟡 5. 没有 smoke test 路径

**根因**：项目没有提供任何 sample/fixture 数据，也没有 `quickstart` 命令。用户无法"先跑起来看看"。

**影响**：新用户的评估路径是：装依赖 → 配网络 → 等同步 → 跑研究 → 看结果。中间任何一步失败都会丢失用户。

### 🔵 6. Agent 子命令在帮助中不可见（已验证无此问题）

`kronos --help` 正常显示 `agent` 子命令组。

### 🔵 7. 无语言选择

**根因**：CLI 帮助和输出全是中文硬编码，没有 i18n 机制。

**影响**：非中文用户完全无法使用。

---

## 根因分析

所有问题的共同根因是：**项目从未被当作"产品"设计过 onboarding 路径**。

Codex 开发时走的是"研究员自己用"的路径——所有外部依赖（API 代理、数据、前置产物）在开发环境中天然存在，不需要解释。但普通用户拿到代码后，这三层依赖全部断裂。

这不是"加个 README"能解决的补丁问题。需要从三个层面做根因修复：

| 层面 | 问题 | 根因修复方向 |
|------|------|-------------|
| **可发现性** | 无 README、无帮助 | 双语 README + CLI 内置 quickstart |
| **可达性** | 数据依赖、配置脆弱 | 内置 sample 数据 + 配置自动发现 |
| **可体验** | Agent 门槛高 | `kronos quickstart` 一键跑通全流程 |
