# Docker 多画像体验修复方案（2026-05-07）

> 来源：`docs/DOCKER_PERSONA_UX_EVALUATION_20260507.md`、`docs/USER_PERSONAS.md`、`docs/PROJECT_STATUS.md`
> 目标版本：v0.4.4
> 范围：只收口 Docker 首次体验的解释方式与引导顺序，不新增模拟盘、实盘、历史重放或新策略类型。

## 一句话结论

v0.4.3 已把 Docker 主路径从“能跑通”推进到“能继续做事”，但还没到“新用户能稳定判断下一步”的程度。

当前剩余问题不是能力缺失，而是第一屏的叙事还没有完全对齐用户画像。最需要收口的三件事是：

1. 结果解释要更短，先让用户知道发生了什么、能不能信、下一步做什么。
2. 策略起草要继续保留内部门槛，但必须先翻成交易语言。
3. Docker 首次构建的噪音要更像进度，不要像故障。

## 主要用户

- L0 / L6：只关心能不能跑起来、结果在哪、下一步做什么。
- L1：想看懂这次结果是不是值得继续。
- L2：有交易想法，愿意继续往下走，但不想先学一套内部术语。
- L3：能接受技术细节，但不希望第一次入口被工程语言打断。

这轮不把 L4 作为主目标。工程接手者仍然重要，但不是第一次安装路径的主阅读人群。

## 问题与根因

### P0：第一屏还是偏长，结果解释没有完全收口

**用户看到的问题**

- `quickstart`、`report latest` 和首次报告之间的解释链路还是偏长。
- 用户能看到结论，但还要自己拼出：用了什么数据、是不是 sample、这次算不算可信、下一步该干什么。
- 当前输出里仍混有安装过程、研究结论和 artifact 路径，读起来像一段混合信息。

**根因**

- 第一屏同时承担安装进度、研究结论和排障信息，职责太多。
- 输出没有一个稳定的结果卡结构，导致不同命令的叙事风格不一致。
- sample 数据与真实数据的边界已经能说清，但还没有被压缩成一眼可读的固定格式。

**产品问题**

- 新用户很难在第一次运行后立即判断“这是正常试跑，还是策略已经被证明可用”。

**修复方向**

- 统一 `quickstart`、`report latest` 和自动研究摘要的第一屏结构。
- 固定成一张结果卡：数据来源 / 样本范围 / 评估对象 / 结论 / 下一步 / 详细报告。
- sample 场景必须明确写“流程试跑，不代表策略有效性”。

### P1：策略起草和 Agent 入口还带着太多内部术语

**用户看到的问题**

- `strategy draft`、`validate`、`smoke-test`、`register` 这些词对交易者不够自然。
- `agent start` 已经更像助手了，但首屏仍有一点“入口菜单”的感觉。
- 当用户输入不完整时，系统拒绝是对的，但提示语还可以更像交易顾问，而不是内部校验器。

**根因**

- 产品已经把门槛建起来了，但门槛名词还是工程语言。
- 当前输出习惯先讲能力名，再讲用户该做什么。
- `agent start` 的首屏还没有完全改成“先看结论，再选动作”的叙事。

**产品问题**

- 用户能继续做事，但要先学会一组内部流程名，这会抬高首次使用门槛。

**修复方向**

- 保留 `validate / smoke-test / register` 这条门槛链，但对外先翻译成：
  - 先检查配置是否完整
  - 再做空跑确认
  - 最后再进入候选池
- `strategy draft` 的输出里，每一步都要有一句交易语言解释。
- `agent start` 首屏先给“我能帮你把策略想法起草成配置”这类结果导向文案，再列操作项。

### P2：Docker 首次构建噪音还会制造焦虑

**用户看到的问题**

- 构建、下载、安装、依赖解析这些信息太密，第一次看很容易被误读成卡住或报错。
- 实际上流程在正常推进，但用户需要更多“这是正常等待”的提示。

**根因**

- 安装遥测和产品输出共用同一条通道。
- 现有日志更像开发调试输出，不像给新用户看的进度说明。

**产品问题**

- 用户会把正常等待误读成异常，第一次体验的信任感会被打断。

**修复方向**

- 默认只保留阶段性进度提示。
- 让日志更像“正在下载 / 正在生成 / 正在整理结果”，而不是内部事件流。
- 详细调试信息保留给显式 verbose 模式。

## 修复原则

1. 第一屏先回答：发生了什么、能不能信、下一步做什么。
2. 面向交易者先讲交易语言，内部术语放第二层。
3. 一个界面只给一个主动作，避免同时抛出多个下一步。
4. Docker 模式下的命令必须可直接复制执行。
5. sample 试跑永远不能被包装成策略有效性结论。

## 方案拆解

| 方向 | 要改什么 | 为什么要改 |
|---|---|---|
| 结果卡统一 | `quickstart`、`report latest`、自动研究摘要共用同一套第一屏字段 | 让用户不必自己拼信息 |
| 策略语义翻译 | 把 `validate / smoke-test / register` 翻成交易者能懂的话，再保留内部术语 | 保留门槛，但降低门槛感 |
| 入口叙事重排 | `agent start` 先说“我能帮你做什么”，再给按钮/选项 | 让入口像助手，不像控制台 |
| 构建噪音降级 | 默认隐藏内部调试噪音，只显示阶段进度 | 减少“这是不是坏了”的误判 |
| 文档同步 | 把本轮解释规则同步到 README / 快速开始说明 | 避免代码改了，文档还在用旧口径 |

## 验收标准

- 新用户在 Docker 首次运行后，能在前 10 行内回答四个问题：
  - 这次跑了什么
  - 用了什么数据
  - 结果能不能当成有效结论
  - 下一步做什么
- `report latest` 第一屏必须包含数据来源、样本范围、评估对象、结论和下一步。
- `strategy draft` 必须先解释为什么要走 `validate → smoke-test → register`，再给命令。
- `agent start` 的第一屏不再像纯菜单，用户能直接进入“描述策略想法，先起草配置”。
- Docker 首次构建的默认输出不再让人误以为程序报错。
- 7 天 sample 场景绝不能出现“90 天验证已完成”或“策略已被证明有效”这类表述。

## 不做的事

- 不新增模拟盘、实盘、历史重放、市场状态分段评估。
- 不把 sample 数据包装成策略有效性结论。
- 不在本轮重构 Agent runtime 或策略系统架构。
- 不把 Docker 首次体验问题改写成 Web 改造任务。

## 实施顺序

1. 先统一 `quickstart` 和 `report latest` 的结果卡。
2. 再改 `strategy draft` 和 `agent start` 的叙事方式。
3. 然后把 Docker 首次构建的噪音降下来。
4. 最后用同一套多画像流程重新跑一遍 fresh clone 评测。
5. 通过之后，再决定是否同步 README / FAQ / 快速开始说明。

## v0.4.4 实施结果

本轮已按上面的修复方向落地为 v0.4.4：

- `quickstart` 自动研究完成后展示与 `report latest` 一致的结果卡。
- `report latest` 第一屏固定为：数据来源 / 样本范围 / 评估对象 / 结论 / 可信度 / 下一步。
- `strategy draft`、`strategy init-r-breaker` 和 Agent 策略起草分支把后续门槛翻译为检查配置、空跑确认、进入候选池。
- Docker entrypoint 在 quickstart 前解释首次运行会准备环境并生成 sample 流程试跑报告；完成后把主下一步收敛到 `report latest`。
- README、CHANGELOG、TODO、PROJECT_STATUS、ROADMAP、产品设计文档和 OpenSpec 已同步到 v0.4.4。

## 已确认的决策点

1. 是否同意把本轮目标锁定为“首次体验语义收口”，而不是继续扩策略能力。
2. 是否同意保留 `validate / smoke-test / register` 这套内部门槛，但对外必须先翻译。
3. 是否同意 Docker 第一屏只保留一个主下一步，不再同时强调多条路径。

本轮实现均按上述三个决策执行；未新增模拟盘、实盘、历史重放或新策略模板。
