# Kronos 模拟用户测试修复方案（2026-05-08）

> 来源：`docs/DOCKER_PERSONA_UX_EVALUATION_20260508.md`
> 目标版本：v0.4.6
> 范围：继续收口首次体验里的表达和引导，不新增模拟盘、实盘或新策略类型。

## 一句话结论

v0.4.5 已经把结果、解释和草案三条链路都接上了，但首次入口对 L0/L6 仍偏长，对 L1 仍偏研究语言，对 L2/L3 则已经基本可用。

这轮修复要做的，不是再加功能，而是把“第一眼看到什么、下一步做什么、什么时候算结论”再压缩一层。

## 主要问题

### P0：第一次安装后的结果判断还不够短

**用户看到的问题**

- 需要在 sample、结果、草案、下一步之间自己再拼一次。
- 读完还要确认“这是不是只是在试跑”。

**根因**

- 结果卡已经统一，但首屏仍同时承担了安装、研究、解释和闸门说明。
- 对 L0/L6 来说，信息密度仍偏高。

**修复方向**

- 把首次路径的主结论压成一句更短的话。
- 在 sample 场景里只保留“试跑 / 不代表结论 / 下一步”三项核心信息。
- 其余细节下沉到 `report latest` 和报告正文。

### P1：`agent start` 还可以更像助手

**用户看到的问题**

- 首屏已经正确，但仍有菜单感。
- 有数据时和没数据时的引导风格还可以更一致。

**根因**

- 当前入口先列选项，再给动作。
- 还没有完全按“先判断结果，再给一个动作”重排。

**修复方向**

- 首屏改成更明确的结果导向叙事。
- 用更少的选项，把默认动作压到最前。
- 对 L0/L1 继续保留自然语言解释，减少内部词。

### P1：`report replay` 的缺内容路径太硬

**用户看到的问题**

- 没有回放报告时，提示比较像工具报错。

**根因**

- 这条命令是下钻入口，不是主入口。
- 缺少内容时没有额外的产品解释。

**修复方向**

- 在没有回放时，补一句“这类报告需要先有回放结果”的产品解释。
- 保留命令名，但把失败提示翻成下一步建议。

### P2：构建和首次等待仍然容易让人误判

**用户看到的问题**

- 首次安装时等待感仍然比较强。
- 首次构建时依赖下载很慢。

**根因**

- Docker 和依赖安装有客观成本，但本轮慢构建不是纯粹的客观成本。
- 首屏缺少更明确的阶段提示。
- Dockerfile 里的 `uv sync --no-cache` 让 fresh build 无法复用依赖层缓存，首次构建会被反复放大。

**修复方向**

- 让阶段提示更像“正在准备 / 正在生成 / 正在分析”。
- 把调试噪音继续压到显式模式里。
- 让 entrypoint 明确提示首次依赖下载是正常流程，减少误判。
- 去掉 `uv sync --no-cache`，让 Docker 依赖层按 `pyproject.toml` / `uv.lock` / `kronos/` 变更正常缓存。

## 已执行修复与验证

- `Dockerfile` 已移除 `uv sync --no-cache`，保留 `uv sync --frozen --no-dev`，让依赖准备回到可缓存路径。
- `docker compose -p kronosfreshv046test4 up --build --abort-on-container-exit` 已通过；quickstart 自动生成 sample 数据、注册 R-breaker、产出报告并正常退出。
- fresh build 证据：修复前 `kronosfreshv046test3` 的 `uv sync` 依赖准备耗时 15m49s；修复后 `kronosfreshv046test4` 的依赖准备耗时 11.99s。
- 针对性回归：`uv run pytest tests/unit/test_dockerfile.py tests/integration/test_cli.py tests/unit/test_reports_latest.py tests/unit/agent/test_console.py -q` 通过，49 passed。
- 静态检查：`uv run ruff check .` 通过；`git diff --check` 通过。

## 修复原则

1. 首屏只回答一个最重要的问题。
2. sample 试跑不被包装成有效性结论。
3. 面向交易者先说交易话，内部词放后面。
4. 没有对应内容时，提示必须告诉用户下一步，而不是只说“找不到”。

## 验收标准

- 新用户在前 10 行内能知道这是试跑还是结论。
- `agent start` 首屏更像一个助手，而不是选项列表。
- `report replay` 缺内容时有明确的产品解释和下一步建议。
- `quickstart` 的等待提示更像进度，不像报错。
