## Context

v0.4.4 已经把首次体验和 `report latest` 的第一屏稳定下来，但用户在下一步仍然会问“为什么会这样”“在哪些市场状态下会这样”“能不能先放到只读观察里再看”。

本变更要把这些问题组织成一条一致的解释路径，而不是再堆一个独立命令。

## Goals / Non-Goals

### Goals

- 让用户能从最新报告继续下钻到关键交易重放。
- 让用户能按市场状态看到独立结论。
- 让用户能把只读观察和实盘执行清楚分开。
- 让多品种策略的 smoke-test 对每个 symbol 都可见、可追踪。

### Non-Goals

- 不做全量 tick 级回放。
- 不做实时下单或真实资金执行。
- 不做自动参数优化。
- 不引入新的 Agent 框架或交易执行框架。

## Decisions

### D1: 关键交易优先于全量回放

**决策**：默认只展示关键交易和关键失败窗口，不做全量逐 bar 播放。

**理由**：

- 产品目标是帮助用户做判断，不是展示大量重复过程。
- 关键节点更能解释策略逻辑和失效模式。

### D2: 市场状态分段要成为标准结果

**决策**：牛市、熊市、震荡市和高波动环境必须是标准分段，而不是附加统计。

**理由**：

- 交易者最关心的不是一个平均值，而是“什么时候能用”。
- 分段结论比总均值更适合作为后续观察门槛。

### D3: 只读观察态必须显式只读

**决策**：只读观察态必须通过 UI、CLI 和状态文案显式区分，不允许和真实执行态共享模糊术语。

**理由**：

- 观察态一旦和实盘态混淆，用户会错误理解风险边界。
- 只读标识比“默认不会执行”更可靠。

### D4: smoke-test 以 symbol 为最小检查单元

**决策**：每个 symbol 都必须独立通过或失败，不能用整体成功掩盖局部缺失。

**理由**：

- 多品种配置的真实风险通常出在个别 symbol。
- 逐 symbol 结果更适合排查数据缺口和配置偏差。

## Risks / Trade-offs

- **[关键交易重放做得太细，用户会觉得信息过载]** -> Mitigation：默认只展示关键节点，完整原始数据保留为下钻。
- **[市场分段阈值不稳定]** -> Mitigation：先固定清晰的状态定义，再逐步做更细粒度切分。
- **[只读观察容易被误读成实盘前置]** -> Mitigation：所有入口都必须显式标注只读、虚拟和人工闸门。
- **[逐 symbol smoke-test 增加运行时间]** -> Mitigation：优先保证可见性和正确性，之后再优化性能。
