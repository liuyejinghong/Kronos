## Context

Kronos 目前能告诉用户“当前只到研究报告，不会启动模拟盘或真实订单”。这解决了边界问题，但还没有解决行动问题：如果用户想继续观察，系统应该给出一份具体计划，而不是让用户自己猜观察规则。

## Goals / Non-Goals

### Goals

- 让用户能从已有报告生成只读观察计划。
- 让计划明确说明是否适合观察。
- 让虚拟订单、延迟、滑点和人工闸门成为报告中的一等信息。
- 为后续实时模拟盘保留可替换接口。

### Non-Goals

- 不做实时行情。
- 不做订单撮合。
- 不做虚拟收益曲线。
- 不接交易所 API Key。
- 不做实盘执行。

## Decisions

### D1: 计划先于引擎

**决策**：v0.4.7 只生成观察计划，不启动模拟盘引擎。

**理由**：

- 当前真正缺的是产品闸门，不是撮合逻辑。
- 计划能先锁定观察假设，避免后续模拟盘实现跑偏。

### D2: 报告驱动，不凭空生成

**决策**：观察计划必须来自最新报告或指定报告。

**理由**：

- 观察计划必须继承样本、策略和验证结论。
- 如果没有报告，系统没有资格建议观察。

### D3: sample 和短样本必须降级

**决策**：sample 试跑和短样本不能进入“建议观察”。

**理由**：

- sample 只能证明流程跑通。
- 短样本容易误导用户把偶然结果当成策略质量。

### D4: 人工闸门写入计划

**决策**：计划必须写明人工确认前不能进入更强执行态。

**理由**：

- 观察态很容易被误解为实盘前置。
- 人工闸门是产品安全边界，不是实现细节。

## Risks / Trade-offs

- **[用户误以为已经启动模拟盘]** -> Mitigation：标题、状态和正文都写明这是计划，不是运行态。
- **[规则过于保守]** -> Mitigation：这是进入实时模拟盘前的闸门版本，宁可保守。
- **[未来模拟盘引擎重复实现计划逻辑]** -> Mitigation：计划产物保留结构化假设，后续可迁移。
