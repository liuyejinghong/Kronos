## Context

Kronos 已经可以把 R-breaker 配置写入候选池，也可以验证、烟雾测试和注册策略。但用户从“有一个交易想法”走到“一个可研究的策略资产”之间，仍然需要手工整理字段、选择模板、补参数和理解下一步命令。

本变更要解决的是这条路径，而不是扩展执行层或重新设计策略引擎。

## Goals / Non-Goals

### Goals

- 把自然语言策略想法转成结构化策略概要。
- 把概要转成可验证的 TOML 草案。
- 清楚区分“支持”“澄清”“拒绝”三种结果。
- 让草案自然接入现有 validate / smoke-test / register 链路。
- 保留 prompt / model / artifact / path 的可追溯性。

### Non-Goals

- 不做任意新策略语言。
- 不做完整代码生成器。
- 不做历史重放。
- 不做实时模拟盘。
- 不做实盘执行。
- 不引入新的 Agent 框架或 LLM 网关。

## Decisions

### D1: 模板驱动，而不是自由生成

**决策**：策略起草必须通过模板驱动的方式落地，首版只允许命中当前支持的策略模板。系统可以解释和推荐，但不能把任何描述都伪造成一个可用策略。

**理由**：

- 现有策略配置模型是有限的，不能假装已经支持任意策略。
- 模板驱动能把自然语言约束在已验证的配置边界内。
- 这样能避免生成一份看似完整、实际无法验证的伪策略。

### D2: 先概要，再草案

**决策**：策略起草必须先输出人能读懂的策略概要，再允许落盘 TOML。

**理由**：

- 用户需要先确认系统理解是否正确。
- 概要可以暴露默认假设、未确定项和风险。
- 这一步可以减少后续反复改 TOML 的成本。

### D3: 不支持就拒绝，不静默补全

**决策**：遇到无法映射到当前支持模板的输入时，系统必须明确拒绝或要求澄清，不允许安静地编出一个“像样”的策略。

**理由**：

- 产品目标是可信，不是显得聪明。
- 静默补全会让用户误以为系统已经理解并实现了他的策略。

### D4: 草案必须继续走现有验证链路

**决策**：策略起草的输出必须继续使用现有的 `validate / smoke-test / register` 流程作为唯一出口。

**理由**：

- 现有链路已经稳定。
- 这样能保持 v0.4.3 和 v0.4.0 的一致性，避免新版本重建一套验证体系。

### D5: 追溯性优先于花哨交互

**决策**：起草过程必须保留 prompt / model / path / artifact traceability，优先让用户和开发者能追踪到这份草案是怎么来的。

**理由**：

- 这类功能容易出现 LLM 偏差，追溯性比界面花样更重要。
- 后续如果要扩展更多模板，追溯性会直接决定调试效率。

## Risks / Trade-offs

- **[模板太少，用户觉得能力弱]** -> Mitigation：明确这是 v0.4.3 的范围，并把 unsupported path 做得诚实、清晰。
- **[LLM 输出过于发散]** -> Mitigation：用结构化概要、模板约束和验证链路三重收口。
- **[把起草误当成验证完成]** -> Mitigation：所有输出都必须把 draft 和 valid 区分开。
- **[用户误以为系统会自动帮他赚钱]** -> Mitigation：严禁在起草阶段输出盈利或有效性结论。
